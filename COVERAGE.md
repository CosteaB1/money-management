# Backend code coverage

This document describes how backend **line** coverage is measured, what is
excluded and why, and the regression gate. Branch coverage is **not** a goal —
we cover every reachable hand-written executable line, but do not chase branch
arms.

## TL;DR

```powershell
# 1. Run tests with coverage (uses coverlet.runsettings).
dotnet test --collect:"XPlat Code Coverage" --settings coverlet.runsettings --results-directory ./CoverageRun

# 2. Build the filtered report (drops generated code; see "Coverage exclusions").
reportgenerator -reports:"./CoverageRun/**/coverage.cobertura.xml" `
  -targetdir:./CoverageReport -reporttypes:"TextSummary;Html" `
  "-classfilters:-System.Text.RegularExpressions.Generated*;-Microsoft.AspNetCore.OpenApi.Generated*;-System.Runtime.CompilerServices.*OpenApi*" `
  "-filefilters:-**/RegexGenerator.g.cs;-**/Migrations/**;-**/*.Designer.cs;-**/*ModelSnapshot*;-**/OpenApiXmlCommentSupport.generated.cs"

# 3. Enforce the line gate (fails the build if total line coverage < threshold).
powershell -NoProfile -File scripts/check-coverage.ps1 -Threshold 99
```

`reportgenerator` ships as a dotnet global tool at `~/.dotnet/tools`; add that to
`PATH`.

## What is measured

Only the five production assemblies (`MoneyManagement.Api`,
`MoneyManagement.Application`, `MoneyManagement.Domain`,
`MoneyManagement.Infrastructure`, `MoneyManagement.SharedKernel`) — scoped via
the `<Include>` block in `coverlet.runsettings`. Test assemblies and third-party
code are not part of the denominator.

## Coverage exclusions (why each is excluded)

Exclusions fall into two buckets: **generated code** (filtered at the report
stage) and **hand-written but genuinely-untestable code** (marked with
`[ExcludeFromCodeCoverage]` at the source, each justified inline).

### Generated code — filtered by `coverlet.runsettings` + reportgenerator filters

| What | How it's excluded | Why |
|------|-------------------|-----|
| EF Core migrations | `ExcludeByFile` `**/Migrations/**` + reportgenerator `-filefilters` | Generated by `dotnet ef migrations add`; not hand-written. |
| EF model snapshot | `**/*ApplicationDbContextModelSnapshot*` / `*ModelSnapshot*` | Generated EF snapshot. |
| `*.Designer.cs` | `**/*.Designer.cs` | Auto-generated migration designer files. |
| `[GeneratedRegex]` output | reportgenerator `-classfilters:-System.Text.RegularExpressions.Generated*` + `-filefilters:-**/RegexGenerator.g.cs` | Source-generator output for the `maib` parser / hex-color / whitespace regexes. The hand-written `[GeneratedRegex]` *declarations* are covered; only the emitted matcher state-machines are filtered. |
| OpenAPI XML-comment generator | reportgenerator `-classfilters:-Microsoft.AspNetCore.OpenApi.Generated*;-System.Runtime.CompilerServices.*OpenApi*` + `-filefilters:-**/OpenApiXmlCommentSupport.generated.cs` | Source-generator output backing Swagger/Scalar docs. |

### Hand-written, genuinely-untestable — `[ExcludeFromCodeCoverage]` attributes

| Member | File | Justification |
|--------|------|---------------|
| `DesignTimeDbContextFactory` (class) | `Infrastructure/Database/DesignTimeDbContextFactory.cs` | Design-time `dotnet ef` tooling entry point; never runs at application runtime. |
| `BnmAutoFetchService.RunRefreshLoopAsync` | `Infrastructure/FxRates/BnmAutoFetchService.cs` | Infinite `Task.Delay(interval)` loop, interval clamped ≥ 1 h, no injectable timer — only reachable via a real wall-clock wait. The backfill path it shares (`DispatchAsync`) is fully covered. |
| `BnmAutoFetchService.RefreshTodayAsync` | same | Only called from the delay-gated loop above. |
| `MigrationExtensions.ApplyMigrations` | `Api/Extensions/MigrationExtensions.cs` | Host-startup migration apply; the pending-migrations path only runs against a fresh live DB. The integration suite points at an already-migrated `money_management_inttest`. |
| `BudgetPeriodGuard.EnsureSucceeded` | `Application/Features/Budgets/BudgetPeriodGuard.cs` | Shared "this can never fail" guard for the budget-period event handlers. Callers only pass domain results guaranteed to succeed (valid year/month from a `DateOnly`, pre-filtered positive amounts); the throw is unreachable by construction. Extracting it here keeps that single defensive throw out of four otherwise-fully-covered handlers. |

## Known uncovered lines (unreachable defensive guards — NOT excluded)

A small residue (~0.8%, ~50 lines) is left uncovered rather than excluded or
gamed. These are defensive guards that cannot be reached without breaking an
upstream invariant, plus three coverlet `const`-declaration artifacts. They are
left visible (not attribute-suppressed) so a future change that *does* make them
reachable will show up as a coverage gap to be tested. They are why the gate
threshold is **99**, not 100.

| Area | Lines | Why unreachable |
|------|-------|-----------------|
| `ArchiveBudgetCommandHandler` / `ArchiveGoalCommandHandler` — `if (archive.IsFailure) return archive;` | 2 + 2 | `Budget.Archive()` / `SavingsGoal.Archive()` always return success (idempotent set-flag). |
| `GetBudgetsQueryHandler.ComputeStatus` — `limit <= 0` arm | 2 | `Budget.Create` guarantees `MonthlyLimit > 0`; persisted rows can't have a non-positive limit. |
| `UpdateTransactionCategoryCommandHandler.IsFlowCompatible` — `_ => false` | 1 | `Category.Create` rejects undefined `CategoryFlow`, so a persisted category's flow is always a defined enum member. |
| `CreateTransferCommandHandler` — destination-leg factory failure | 2 | Same-currency: the destination money equals the (already-validated) source money. Cross-currency: the destination amount is validated `> 0` earlier. The factory therefore can't fail once the source leg succeeded. |
| `CommitImportCommandHandler` — matching counter-leg factory failure | 2 | Same reasoning as the transfer destination leg: counter amount is validated/derived from already-valid inputs. |
| `UpdateGoalCommandHandler` — `modeSwitch.IsFailure` | 2 | `LinkAccount` is only reached after an account-exists check (a real, non-empty id); `Unlink()` always succeeds. |
| `UpdateManualSavedCommandHandler` — `contribution.IsFailure` | 2 | The contribution amount is non-zero (`delta != 0` guard), MDL, dated today, no notes — every `SavingsGoalContribution.Create` guard passes. |
| `GetBalanceOverTimeQueryHandler.BuildAsOfDates` — Monthly final `dates.Add(to)` | 3 | The month-walk loop always clamps its final month-end to `to`, so `dates[^1] == to` already. |
| `GetGoalDetailQueryHandler` — `BuildMonthEndPoints` final append, `BuildSavedHistory` close-on-today append, `DedupeByAsOf` overwrite, `BuildPaceStats` `monthsToAchieve <= 0`, `ComputeLinkedAvgMonthly` early returns; plus 3 `const` declarations | 15 | Same month-end-clamp invariant as above; `monthsToAchieve` is strictly positive when `saved < target` and `avg > 0`; the 3 `const decimal` lines are a coverlet sequence-point artifact (consts have no runtime initializer). |
| `EfBackupStore.InsertSql` — non-empty schema branch | 3 | No entity is mapped to a non-default Postgres schema, so `GetSchema()` is always null/empty. |
| `CategorySeeder` / `CategoryPatternSeeder` — `result.IsFailure` skip-and-log | 3 + 4 | The seed `Defaults` are hardcoded valid names/keywords, so `Category.Create` / `CategoryPattern.Create` never fail for them. |
| `DataEndpoints` (export failure, > 50 MB), `ImportEndpoints` (> 5 MB), `ReportsEndpoints` (CSV export failure), `GlobalExceptionHandler` (raw `JsonException` arm) | ~10 | Export query handlers don't fail; the file-size guards need multi-megabyte uploads; minimal-API binding wraps `JsonException` inside `BadHttpRequestException` (the covered arm), so the bare `JsonException` case isn't reached in practice. |

If you make any of these reachable, add a real test — do not extend the
exclusion list to hide it.

## The gate

`scripts/check-coverage.ps1` parses `CoverageReport/Summary.txt` (the filtered
TextSummary) and fails (non-zero exit) when total **line** coverage is below
`-Threshold` (default 99). It ignores branch coverage by design. Wire it as the
last step of the coverage job so a coverage regression breaks the build.
